# 对TcpQuality Canvas的说明
使用了https://tcpquality.ibsgss.uk/run 项目已经做好的API；并深度修改原作。

## 一、 探测机制的从 Raw Socket 到标准 L4/L7 伪装

* **传统 `tcpping`（图 1 的 `nping`）：** 绕过系统网络栈手工画 TCP 包，引发内核回复 `RST`，被 CDN 防火墙视为攻击直接 Drop（丢弃）。

改用 `curl` 启动标准套接字（Standard Socket）。并且在主循环中**全量使用 `curl` 进行 TCP 连接状态提取**：

* **特点 1：特种 DPI 指纹伪装。** 它注入了真实的桌面 Chrome 浏览器 `User-Agent`，在 CDN 边缘节点的 WAF（Web 应用防火墙）看来，这完全是**合法的用户网页浏览流量**，直接放行。

* **特点 2：“容错算法”。** 对 `curl` 状态码的处理把 `52`（服务器无响应）、`56`（接收数据失败）、`35`（SSL 握手失败）等**全部判定为“探测成功（is_ok=1）”**！

* **为什么？** 因为作为 `tcpping`，脚本只需要知道 **Layer 4（TCP 层）是否握手成功**（通过 `%{time_connect}` 拿到了 RTT）。即使后续由于没有发真实的 HTTP 请求导致 CDN 报错断开，但只要 TCP 握手完联通了，就证明网络没丢包。这种“只要前端过了，后端报错也算过”的逻辑，完美兼顾了 TCP 探测的准确性与 CDN 的特殊响应机制。

## 二、 针对“中国移动（CMI）”与防刷策略的生存优化


国内三网中，中国移动（AS9808/CMI）和中国电信的云机房高防对连续高频的相同频率探测极为敏感。脚本对此做了极其高明的反频率识别（Anti-Rate Limiting）改造：

* **动态阶梯控频（Stagger 错峰）：**

在多线程并发启动时，每个节点会根据索引自动错开零点几秒，**打破并发特征**，防止瞬间流量并发触发 CDN 机房的 DDoS 阈值。

* **移动节点特异性加时与随机扰动（Jitter）：**

如果目标是移动节点，脚本会**强制将探测间隔翻倍**，并且**加入 `RANDOM % 2` 秒的随机时间抖动**。这样，发往移动 CDN 节点的 TCP 请求在时间序列上是完全随机、不连续的，完美混淆了移动防火墙的“固定频率扫射检测”算法，从而彻底绝育了“假丢包”。

## 三、 宿主机保护-并发熔断



针对单核 512MB-1G 内存的“弱鸡”。如果开局直接并发轰炸 40-50 个线程，单核 CPU 会瞬间被调度锁死（上下文切换开销过大），导致严重的**应用层内部丢包错觉**。


* **改造点：** 脚本锁死了 `MAX_PREWARM_JOBS=15`。在预热阶段，通过检测 `jobs -pr | wc -l` 严格控制后台进程不超标。

* **内存与僵尸进程全清理：** 挂载了硬核的 `trap cleanup_all_processes EXIT SIGINT SIGTERM`，使用 `kill -9 $pids` 确保一旦用户 Ctrl+C 退出或 120 秒超时熔断时，**干净利落地掐灭所有底层 curl 异步进程**，不留内存泄漏或僵尸进程。
